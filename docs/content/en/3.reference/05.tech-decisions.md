Conscious technical decisions and their rationale.

## `trns.list` / `trns.delta` - Convex pagination

`convex/trns.ts` uses Convex's built-in `.paginate()` API. The frontend helper `fetchAllPages()` iterates through all pages to collect the full result.

**Why pagination instead of JSON.stringify:**
Previously, queries used `JSON.stringify` to bypass the Convex 8,192 array element limit. Pagination is the standard Convex approach - it avoids serialization overhead, provides proper typing, and handles large datasets natively without workarounds.

## Incremental sync (delta)

`initTrns()` uses incremental sync instead of full reload on every open. Details: [Incremental Sync](/en/development/sync).

**Why not Convex reactive queries:**
Convex supports real-time subscriptions, but they require a constant WebSocket connection and resend the full query result on any change. For 10K+ transactions this is inefficient. Delta sync fetches only changed records.

**Why hash mismatch → fullSync instead of ids query:**
Previously, deletion detection used a separate `trns.ids` query to fetch all IDs and find deletions. This query was broken at 8,000+ trns (hit the 8,192 array limit). The new approach: after applying delta, compute a local XOR hash and compare with the server hash. On mismatch, fall back to `fullSync()`. Deletions are rare (1-2 per month), so a one-time full sync is an acceptable trade-off - simpler and more reliable than maintaining a paginated IDs query.

**Why not soft-delete:**
Soft-delete (`deletedAt` field) would require migrating all existing data, modifying all queries to filter deleted records, and periodic cleanup. Hash-based detection is simpler and requires no schema changes.

**Why delta sync only for transactions:**
Categories and wallets are small datasets (tens to hundreds of records). Full sync is fast enough. Transactions are the only entity that grows to thousands of records.

## Optimistic UI with fire-and-forget mutations

Store methods update the UI immediately, then fire Convex mutations without awaiting. The [offline queue](/en/development/offline-queue) captures every operation, so nothing is lost even if the mutation fails.

**Why fire-and-forget:**
Awaiting mutations would block the UI and make the app feel slow. Optimistic updates give instant feedback. The offline queue + replay mechanism ensures eventual consistency.

**Why `shallowRef` for store items:**
All Pinia stores use `shallowRef` instead of `ref` for the items map. This avoids deep reactivity tracking on potentially thousands of objects, significantly improving performance. Store methods create new object references on mutation to trigger reactivity.

## Frontend ID system

New entities get temporary IDs with `local_` prefix (e.g., `local_a1b2c3`). After successful server creation, the ID is remapped to a Convex ID. Full lifecycle details: [Offline Queue - Frontend ID System](/en/development/offline-queue#frontend-id-system).

**Why `local_` prefix:**
Explicit prefix makes intent clear in code - `isLocalId(id)` checks `id.startsWith('local_')`. No dependency on Convex ID format internals.

**Why not UUID:**
A short random suffix is sufficient for a single-user app. UUIDs are longer and harder to read in logs.

## Hard navigation on sign-out

Sign-out uses `window.location.href = '/login'` instead of Vue Router navigation.

**Why:**
Hard navigation destroys all JavaScript state - Pinia stores, WebSocket connections, reactive watchers, cached data. This prevents stale auth state from leaking into the next session. Vue Router `navigateTo()` would keep all in-memory state alive.

## Safe arithmetic parser in calculator

The transaction form calculator uses a recursive descent parser (`app/components/trnForm/utils/calculate.ts`) instead of `eval()` or `new Function()`.

**Why:**
`new Function(expression)` is a code injection vector - a malicious string could execute arbitrary JavaScript. It also requires `unsafe-eval` in Content-Security-Policy. The custom parser supports only `+`, `-`, `*`, `/` on numeric literals, rejecting everything else.

## Cascade deletes in Convex

Convex has no foreign key constraints, so cascade deletes are manual - `wallets.remove` and `categories.remove` delete all associated transactions in the same mutation. The frontend sends only the parent entity delete to the [offline queue](/en/development/offline-queue#backend-cascade).

**Why paginated batches (500 items):**
Convex limits documents written per transaction to 16,000. A wallet with thousands of transactions could exceed this limit in a single mutation. Cascade deletes now paginate: fetch a batch of 500 transaction IDs, delete them, repeat until none remain, then delete the parent entity.

## Computed children instead of denormalized `childIds`

Categories store only `parentId`. Child IDs are computed dynamically by filtering categories where `parentId === id`.

**Why:**
Previously, parent categories stored a `childIds` array. This required keeping both sides in sync on every reparent, create, or delete - a source of bugs. Computing children from `parentId` is a single source of truth with no sync issues. The category count is small enough (tens to hundreds) that the lookup cost is negligible.

## Category nesting limited to 2 levels

Categories support only parent → child, not deeper nesting.

**Why:**
Deeper nesting complicates the UI (statistics breakdown, category picker, breadcrumbs) without adding practical value for personal finance tracking. Two levels cover 99% of use cases (e.g., Food > Groceries).

## Theme injection priority

Theme styles use `tagPriority: -2` in `useHead()`, while plugin scripts use `tagPriority: -1`.

**Why:**
This ensures theme CSS variables are applied before the first paint. The priority chain: `useHead` styles (-2) → plugin scripts (-1) → `@layer theme` (always loses to non-layered). Non-layered CSS in `theme.css` would override `useHead()` by document order, which is why `--ui-radius` and `--ui-primary` must not be set there.

## Convex limits

Reference ([docs](https://docs.convex.dev/production/state/limits)):
- Function return value: 16 MiB
- Data read per transaction: 16 MiB
- Documents scanned per query: 32,000
- Document size: 1 MiB
- Array elements per document: 8,192
- Documents written per transaction: 16,000
- Query execution time: 1 sec (user code, excluding DB ops)
