## Current approach: dual validation

The project uses two independent validation systems:

| Layer | Framework | Purpose |
|-------|-----------|---------|
| **Client** (browser) | Zod v4 | Form UX - defaults, trimming, instant error feedback |
| **Server** (Convex) | `convex/values` (`v.*`) | Data integrity - type-safe ID refs, ownership checks |

### Where validation lives

| Entity | Convex validators | Zod schemas |
|--------|-------------------|-------------|
| Wallets | `convex/wallets.ts`, `convex/schema.ts` | `app/components/wallets/types.ts` |
| Categories | `convex/categories.ts`, `convex/schema.ts` | `app/components/categories/types.ts` |
| Transactions | `convex/trns.ts`, `convex/schema.ts` | `app/components/trns/types.ts` |
| User Settings | `convex/userSettings.ts` | - (no client schema) |
| Rates | `convex/rates.ts` | - (no client schema) |
| Stat config | - | `app/components/stat/useStatConfig.ts` |
| Date params | - | `app/components/date/useStatDate.ts` |

### What each side validates

**Zod (client-only):**
- `.trim().min(1)` on name/icon - prevents empty after trim
- `.positive()` on amounts
- `.default()` for form initial values
- `.transform()` for query param parsing (string → number)

**Convex (server-only):**
- `v.id('wallets')`, `v.id('categories')` - type-safe ID references
- Entity ownership (`userId` match)
- Category circular reference prevention
- Category name uniqueness per parent
- Category nesting depth limit (2 levels)
- Transfer wallet difference check
- String length limits: name (100), description (500), icon (50), color (50), currency (10)
- Numeric ranges: amount (max 1 trillion), creditLimit (max 1 billion), date (2000–2099)
- `categoryId` enforcement: must be `'transfer'` for Transfer type, cannot be `'transfer'` for non-Transfer

**Both sides (redundant ~40%):**
- Field presence / optionality
- Basic type checks (string, number, boolean)
- Union / literal validation (TrnType, WalletType enums)

### Why constants are duplicated

`TrnType` enum and `walletTypes` array are defined independently on each side:
- `convex/validators.ts` - for server (imports `v` from `convex/values`)
- `app/components/trns/types.ts` / `app/components/wallets/types.ts` - for client

Importing from `convex/` directory into browser code pulls in server-only modules (`_generated/server`, auth) and triggers "Convex functions should not be imported in the browser" warning. See [tech-decisions](/en/development/tech-decisions) for details.

## Evaluated alternatives

### fluent-convex + shared Zod schemas

[fluent-convex](https://github.com/mikecann/fluent-convex) (v0.x) provides a fluent API for Convex functions with a Zod plugin that auto-converts Zod schemas to Convex validators.

**Would enable:** single Zod schema used on both client and server.

**Rejected because:**
- Library is experimental (v0.x, single author, 90 stars)
- `v.id('wallets')` has no Zod equivalent - loses type-safe ID references
- `schema.ts` (table definitions) must stay on `v.*` regardless
- Extra dependency: `convex-helpers` as peer dep

### Shared Zod schemas with manual convex-helpers conversion

Define Zod schemas in a `shared/` directory. Client uses them directly. Server converts via `convex-helpers/server/zod`.

**Would enable:** single source of truth without fluent-convex dependency.

**Rejected because:**
- Same `v.id()` limitation
- Requires configuring Convex tsconfig to import from outside `convex/`
- Boilerplate for manual Zod → Convex conversion
- Marginal benefit for 3 stable entities with ~15 fields each

## Decision

Keep dual validation. The duplication is small (~15 fields × 3 entities), schemas change rarely, and each side serves a distinct purpose:
- **Zod** = UX (defaults, transforms, instant feedback)
- **Convex** = integrity (ID refs, ownership, business rules)

### When to reconsider

- `convex-helpers/server/zod` gaining native `v.id()` support
- fluent-convex reaching v1.0 with stable API
