Transactions use incremental (delta) sync to avoid re-downloading all data on every app open. Called from `initTrns()` during [app initialization](/en/development/architecture#app-initialization).

## How It Works

1. Read `syncMeta` from LocalForage
2. **No syncMeta or no cache** → `fullSync()`: fetch all via paginated `trns.list`, save syncMeta
3. **Has syncMeta + cache** → `deltaSync()`: fetch only changes via paginated `trns.delta` + `trns.idsHash` (parallel)
4. On delta error or hash mismatch → fallback to `fullSync()`
5. Both paths end with `mergeOfflineOps()` + `setTrns()`

## Convex Queries

| Query | Args | Returns | Index |
|-------|------|---------|-------|
| `trns.list` | `paginationOpts` | Paginated user trns (`PaginationResult`) | `by_user_date` |
| `trns.delta` | `paginationOpts`, `since: number` | Paginated trns with `updatedAt > since` (`PaginationResult`) | `by_user_updatedAt` |
| `trns.idsHash` | - | XOR hash of all trn IDs (string) | `by_user` |

## Sync Metadata

Stored in LocalForage under key `finapp.trns.syncMeta`:

```typescript
type TrnsSyncMeta = {
  lastSyncedAt: number   // timestamp of last successful sync
  idsHash: string        // XOR hash of all trn IDs at last sync
}
```

## Deletion Detection

Since Convex uses hard deletes (no `deletedAt` field), deletions are detected by hash comparison:

1. `deltaSync()` applies delta changes to cached data
2. Computes local XOR hash from all non-local IDs in the resulting data
3. Compares with `trns.idsHash()` from server
4. **Hash match** → no deletions, sync complete
5. **Hash mismatch** → deletions or missed changes detected → fallback to `fullSync()`

This approach is simple and reliable. Deletions are rare (1-2 per month), so a one-time `fullSync()` on hash mismatch is an acceptable trade-off.

## Full Sync Triggers

Full sync runs when:
- First app open (no syncMeta in LocalForage)
- LocalForage cache is cleared
- Delta sync fails (network error, corrupt data)
- Hash mismatch after applying delta (deletions detected)

## Interaction with Offline Queue

Delta sync is fully compatible with the [offline queue](/en/development/offline-queue):
- `mergeOfflineOps()` runs after both full and delta sync
- `cleanupFrontendIds()` removes stale frontend IDs from the merged data that are no longer in pending ops (already synced in a previous session but still cached)
- Offline-created trns (frontend IDs) are not affected by deletion detection (only Convex IDs are compared)
- The offline queue replay happens after `initTrns()` completes

## Performance

| Metric | Full sync (10K trns) | Delta sync (10 changes) |
|--------|---------------------|------------------------|
| Network | ~2 MB (paginated) | ~2 KB (delta) + hash |
| Parsing | 10K objects | 10 objects |
| Reduction | - | ~99% network and parsing |
