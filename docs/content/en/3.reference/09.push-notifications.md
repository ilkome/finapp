## Overview

Finapp delivers notifications via **Web Push (VAPID)** through the existing PWA
service worker - no native wrapper, no FCM/APNs. One VAPID keypair signs every
push; the public key ships to the client, the private key is a Supabase function
secret.

```
client (subscribe)            Supabase                       push service
─────────────────             ────────                       ────────────
pushManager.subscribe ──► push_subscriptions (RLS)
                                   ▲
send-push (service role) ──────────┘ reads subs ──► web-push ──► browser ──► SW push event
```

## Platform support

- **Android (Chrome)**: works in the browser and the installed PWA.
- **iOS (Safari 16.4+)**: Web Push works **only** in a PWA installed to the home
  screen. The permission prompt must follow a user gesture. The settings toggle
  shows an install hint when running in a non-standalone iOS tab.

## Pieces

| Piece | Location |
| --- | --- |
| SW handlers (`push`, `notificationclick`) | `app/public/sw-push.js`, injected via `workbox.importScripts` |
| Public key | `runtimeConfig.public.vapidPublicKey` (`VITE_VAPID_PUBLIC_KEY`) |
| Subscriptions table | `public.push_subscriptions` migration |
| Client lifecycle | `app/composables/usePushNotifications.ts` + `components/notifications/Settings.vue` |
| Sender | `supabase/functions/send-push` |

## Data model

`push_subscriptions` is **server-side only** - deliberately excluded from the
`powersync` publication and sync rules, so it never reaches client SQLite. The
client writes/removes its own rows via supabase-js (PostgREST), guarded by the
`push_subscriptions_owner` RLS policy. The endpoint column is unique so the
client upserts on re-subscribe.

## send-push function

`POST /functions/v1/send-push` with `{ userId?, title, body?, url?, icon? }`.

Authorization (the gateway already requires a valid JWT):

- A **service_role** JWT may target any `userId` in the body (used by cron jobs
  and server-side triggers).
- A **user** JWT may only target itself; the body `userId` is ignored.

The function reads the target user's subscriptions with the service role
(bypasses RLS), sends each via web-push, and prunes endpoints the push service
reports as gone (`404`/`410`).

Secrets: `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`.

## Test send

```bash
curl -X POST "https://<project>.supabase.co/functions/v1/send-push" \
  -H "Authorization: Bearer <service_role_key>" \
  -H "Content-Type: application/json" \
  -d '{"userId":"<uid>","title":"Finapp","body":"Hello"}'
```

A user must enable notifications in Settings first (which stores a subscription);
otherwise the call returns `{ "sent": 0, "note": "no subscriptions" }`.

## Follow-up

Event triggers (spend reminders via `pg_cron`, budget alerts, etc.) call
`send-push` and are layered on top of this infrastructure.
