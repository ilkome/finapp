Unified ordered offline queue with operation collapsing. All pending operations (create/update/delete) for wallets, categories, and trns are stored in a single ordered array in localforage under key `finapp.offline.queue`.

## Architecture

### Key Files

| File | Description |
|------|-------------|
| `app/components/offline/types.ts` | Types: `OfflineOp`, `EntityType` |
| `app/components/offline/helpers.ts` | Queue API: push, remove, get, clear |
| `app/components/offline/replay.ts` | Replay queue on app restart |
| `app/components/offline/storageKeys.ts` | LocalForage keys for entity caches |

### Collapsing Rules

When pushing a new operation, the queue finds the last operation with the same `entity + id` and collapses:

| Existing | New | Result |
|----------|-----|--------|
| create | update | create (merged data) |
| create | delete | remove both (never reached server) |
| update | update | update (merged data) |
| update | delete | replace with delete |
| delete | * | ignore new (delete is terminal) |
| - | * | append as-is |

### Serialization

`pushOfflineOp` uses a promise chain to serialize concurrent writes - no `await` required at call sites.

### Backend Cascade

Both `wallets.remove` and `categories.remove` on the Convex backend cascade-delete all associated trns. The frontend does NOT push separate trns delete operations when deleting a wallet or category - only the parent entity delete goes into the queue. `removeTrnsFromStore()` handles optimistic UI cleanup without mutations.

## Frontend ID System

New entities need IDs before the server responds. The frontend generates temporary IDs that are later remapped to real Convex IDs.

### Generation

`generateId()` (`utils/generateId.ts`) produces IDs with `local_` prefix + 6 random base-36 characters. Example: `local_a1b2c3`.

### Distinguishing from Convex IDs

`isLocalId(id)` (`app/utils/convexId.ts`) checks `id.startsWith('local_')`. Any ID with the `local_` prefix is a frontend ID.

This check is used throughout the codebase:
- Stores: determine create vs update (`!isLocalId(id) && id in items`)
- Delta sync: exclude local IDs from deletion detection
- Batch delete: filter out local IDs for server mutation
- Replay: skip delete operations for local IDs (nothing to delete on server)

### ID Remapping

After a successful create mutation, the server returns the new Convex ID. `handleMutationResult()` (`app/composables/useStoreSync.ts`) replaces the frontend ID key in the store map:

```
items[frontendId] → items[convexId]  (same entity data, new key)
```

The remapped store is persisted to localforage immediately.

### Stale ID Cleanup

`cleanupFrontendIds()` (`app/utils/convexId.ts`) runs during `mergeOfflineOps()` on every init. It removes local IDs from cached data that are no longer in the pending offline queue - these are IDs from a previous session that were either successfully remapped or abandoned.

### Where Frontend IDs Are Created

| Entity | File | Code |
|--------|------|------|
| Transaction | `app/components/trnForm/useTrnsFormStore.ts` | `id: values.trnId ?? generateId()` |
| Category | `app/components/categories/Form.vue` | `editCategoryId = props.categoryId ?? generateId()` |
| Wallet | `app/components/wallets/Form.vue` | `editWalletId = props.walletId ?? generateId()` |

### Edge Cases

**Parallel creates:** Each remapping is independent - destructuring extracts the specific frontend ID and rebuilds the object. No conflicts between concurrent create mutations.

**Crash before remapping:** If the app crashes after mutation succeeds but before `.then()` runs, the next session's `cleanupFrontendIds()` removes the stale frontend ID from cache, and the entity exists on the server with its Convex ID.

## Scenarios

### Create

#### Online
1. User creates entity (wallet/category/trn)
2. Store generates frontend ID (e.g. `local_abc123`)
3. Optimistic UI: entity appears in store immediately
4. `pushOfflineOp({ type: 'create', ... })` saves to queue
5. Convex mutation fires (fire-and-forget)
6. `.then()`: removes op from queue, remaps frontend ID to Convex ID in store

#### Offline
1. Steps 1-4 same as online
2. Mutation fails (network error) - `.catch()` shows toast
3. Operation stays in queue
4. Entity visible in UI with frontend ID

#### Restart after offline create
1. `initTrns` fetches server data (full sync on first load, [delta sync](/en/development/sync) on subsequent loads); `initWallets/initCategories` fetch full data
2. Pending ops from queue merged on top (frontend ID entries added to store)
3. `replayOfflineQueue()` runs:
   - Frontend ID + create/update → dispatches to store method → mutation fires → `.then()` remaps ID
   - Frontend ID + delete → skipped (nothing to delete on server)
4. Entity gets real Convex ID

### Update

#### Online
1. User edits entity
2. `isExisting = !isLocalId(id) && id in items` → `true`
3. Optimistic UI: store updated immediately
4. `pushOfflineOp({ type: 'update', ... })` - collapses with existing ops
5. Convex `update` mutation fires
6. `.then()`: removes op from queue

#### Offline
1. Steps 1-4 same as online
2. Mutation fails - op stays in queue
3. Multiple edits collapse: update + update → single update with merged data

#### Restart after offline update
1. Init fetches server data - old version (via [delta sync](/en/development/sync) or full sync)
2. Pending update merged on top → UI shows latest version
3. Replay dispatches update → mutation fires → synced

### Delete

#### Online
1. User deletes entity
2. Optimistic UI: removed from store immediately
3. For wallets/categories: `removeTrnsFromStore()` removes child trns from UI
4. `pushOfflineOp({ type: 'delete', ... })` - collapses with existing ops
5. Convex `remove` mutation fires (cascade-deletes trns on backend)
6. `.then()`: removes op from queue

#### Offline
1. Steps 1-4 same as online
2. Mutation fails - op stays in queue
3. Collapsing: if entity was created offline and then deleted → both ops removed (never reached server)

#### Restart after offline delete
1. Init fetches server data - entity still exists on server (via [delta sync](/en/development/sync) or full sync)
2. Pending delete removes entity from merged data
3. Replay dispatches delete → mutation fires → synced

### Edge Cases

#### Create + Edit offline, then restart
1. Create: `pushOfflineOp({ type: 'create', data: v1 })`
2. Edit: collapses to `create` with merged data (v1 + v2)
3. Restart: single create op replayed → server gets final version

#### Create + Delete offline, then restart
1. Create: `pushOfflineOp({ type: 'create', ... })`
2. Delete: collapses → both ops removed from queue
3. Restart: nothing to replay, entity never existed on server

#### Delete wallet/category with trns offline
1. `pushOfflineOp({ type: 'delete', entity: 'wallets', id })` - only wallet delete in queue
2. `removeTrnsFromStore(trnsIds)` - optimistic UI cleanup, no queue ops for trns
3. Restart: wallet delete replayed → backend cascade deletes trns

#### Orphaned trns on replay
1. Trn created offline referencing wallet/category
2. That wallet/category deleted (also offline or by another device)
3. Replay detects orphan: wallet/category no longer in store
4. Orphaned trn op removed from queue, toast shown to user

#### Concurrent pushOfflineOp calls
Promise chain serialization ensures writes don't overwrite each other:
```typescript
let writeChain = Promise.resolve()
export function pushOfflineOp(op) {
  writeChain = writeChain.then(() => impl(op), () => impl(op))
  return writeChain
}
```

#### isReplaying flag
During replay, store methods skip `pushOfflineOp` to avoid re-queuing operations that are already being replayed. The flag is set before the replay loop and cleared in `finally`.

## Known Limitations

### Wallet order not in offline queue

`saveWalletsOrder` uses fire-and-forget without the offline queue. If the user reorders wallets while offline and closes the app before the mutation succeeds, the order is lost on next load (server returns old order, no pending op to merge).

This is intentional - wallet reordering is rare and the added complexity of storing order operations in the queue (special `id: 'order'`, separate merge logic in `initWallets`, extra replay branch) is not justified.
