## App Initialization

```
app starts
├── Plugins
│   ├── theme.ts (enforce: 'post')
│   │   └── Read theme from localStorage → update appConfig
│   └── convex.ts
│       └── Create WebSocket client → setup auth
├── app.vue
│   ├── useHead() - theme CSS vars (tagPriority: -2)
│   ├── useGuard() - auth redirect logic
│   └── render layout + page
└── useInitApp (on authenticated page)
    ├── loadDataFromCache() - parallel localforage reads (6 items)
    ├── loadDataFromDB()
    │   ├── await waitForConvexAuth() - ensure WebSocket is authenticated
    │   ├── parallel store init (5 stores)
    │   │   ├── initCurrencies()
    │   │   ├── initUserSettings()
    │   │   ├── initWallets()
    │   │   ├── initCategories()
    │   │   └── initTrns() - delta or full sync
    │   └── replayOfflineQueue() - replay pending offline ops
```

## Project Structure

```
app/           # Nuxt app: components, composables, pages, plugins
convex/        # Backend: schema, queries, mutations, auth, Telegram bot
services/      # Client-side infrastructure (Convex client factory, API mappers)
utils/         # Pure helpers (id, logger, types)
utils/migrate/ # Firebase → Convex migration scripts (data.json gitignored)
i18n/locales/  # en-US.js, ru-RU.js
```

Conventions worth knowing (not obvious from `tree`):
- `convex/_generated/` is produced by Convex CLI - never edit.
- `services/convex/` is separate from `convex/` on purpose: `convex/` is the deployed backend, `services/` is frontend-only glue (client factory, doc → map converters).
- Each feature under `app/components/<feature>/` owns its Pinia store, form, list, and types - there is no separate `stores/` directory.

## Store Pattern

All Pinia stores (`useTrnsStore`, `useWalletsStore`, `useCategoriesStore`) follow the same pattern:

```
items: shallowRef          # reactive state (shallow for performance)
init()                     # fetch from Convex → mergeOfflineOps → set
set(data)                  # items.value = data + debouncedPersist (300ms)
save({ id, values })       # optimistic UI → pushSaveOp → fire mutation
delete(id)                 # optimistic UI → pushDeleteOp → fire mutation
```

### Data Flow (save)

```
user action
  → save({ id, values })
    → isExisting = !isLocalId(id) && id in items
    → optimistic: set({ ...items, [id]: values })
    → pushSaveOp({ entity, id, isDemo, isExisting, values })
    → fire mutation (no await)
      → .then(): removeOfflineOp + remap frontend ID
      → .catch(): show error toast
```

### Data Flow (init)

```
init()
  → fetch from Convex (query)
  → convert to map (strip _id, _creationTime, userId)
  → mergeOfflineOps(data, entity)
    → read pending ops from queue
    → apply create/update ops on top of server data
    → remove stale frontend IDs
  → set(data)
```

## Convex Client Access

| Composable | Returns | Purpose |
|------------|---------|---------|
| `useConvexClient()` | `ConvexClient` | WebSocket client singleton |
| `useConvexClientWithApi()` | `{ api, client }` | Client + auto-generated `api` object for typed queries/mutations |

Stores use imperative `client.query()` and `client.mutation()` because they manage their own caching and offline merge logic.

**Auth flow (client - `plugins/convex.ts`):**
1. On plugin init: if `hasAuthCookie()` && online → eagerly call `client.setAuth(fetchToken)` to start token fetch in parallel with session resolution
2. `fetchToken` uses `authClient.convex.token()` (Better Auth `convexClient()` plugin) - the `crossDomainClient` plugin handles cookie forwarding via `localStorage['better-auth_cookie']`
3. `authReadyPromise` resolves when first token fetch completes (success or failure) - `loadDataFromDB()` awaits this before issuing Convex queries
4. Watch session state: only act on definitive results (user found → `setAuth()`, user absent without error → `clearAuth()`). Network errors and `isPending` states are ignored to prevent offline logout
5. `online` event: re-creates `authReadyPromise` and calls `setAuth()` for a fresh token
6. `ensureConvexAuth()`: called from `/auth/callback` to set auth BEFORE SPA navigation to dashboard

## Frontend ID System

New entities get temporary IDs with `local_` prefix (e.g., `local_a1b2c3`). `isLocalId(id)` checks `id.startsWith('local_')` to distinguish them. See [rationale](/en/development/tech-decisions#frontend-id-system).

Frontend IDs are remapped to Convex IDs in `handleMutationResult()` after successful create mutations. The store replaces `items[frontendId]` with `items[convexId]`.

## Auth Guard

### Auth Layers

Auth uses two layers:

1. **Route middleware** (`auth.global.ts`): checks `finapp.localAuthUid` cookie to decide whether to allow navigation or redirect to `/login`. Skips network calls when offline, trusting the cached cookie.
2. **Client guard** (`useGuard()` in `app.vue`): watches the Better Auth session asynchronously and handles true session expiry after initial load. On unexpected session loss (not via sign-out), the `finapp.localAuthUid` cookie is cleared.

### OTT Auth Flow

Google OAuth uses a one-time token (OTT) for cross-domain authentication:

1. Google OAuth redirects back with `?ott=...` query parameter
2. Global middleware intercepts any route with `ott` in the query and redirects to `/auth/callback`
3. The callback page verifies the OTT with the Convex Better Auth backend
4. On success: sets auth cookie, calls `ensureConvexAuth()` to start Convex auth immediately, then navigates via `navigateTo()` to the original redirect destination

### Redirect Protection

`getSafeRedirectPath()` (`app/utils/redirect.ts`) validates `?redirect=` query parameters - only relative paths starting with `/` (not `//`) are allowed, preventing open redirect attacks.

### Guard Logic

`useGuard()` in `app.vue` watches `currentUser`:
- Logged in + on `/login` → redirect to `/dashboard` (or `?redirect=` path)
- Not logged in + not on `/login` → redirect to `/login`
- `isSigningOut` flag prevents redirect loop during sign-out

## Offline / Online Handling

The app preserves auth state and data when offline:

- **Auth cookie preserved offline:** Session watch ignores network errors (`s.error`) - only definitive "no user" responses clear the cookie. This prevents logout when offline or on flaky connections.
- **Token fetch skipped offline:** `fetchToken` checks `!navigator.onLine` before making HTTP requests - avoids noisy errors in the console.
- **Reconnect:** `online` event listener re-creates `authReadyPromise` and calls `setAuth()` so that `loadDataFromDB()` (which awaits `waitForConvexAuth()`) gets a fresh token.
- **Route middleware:** `auth.global.ts` checks `hasAuthCookie()` synchronously - no network calls, works offline.
- **PWA:** Service worker precaches all build assets + `NavigationRoute` serves SPA shell for all navigation requests.

## Sign-Out Flow

```
signOut()
  → isSigningOut = true (prevents auth guard redirect loop)
  → clear store data (trns, categories, wallets, user)
  → clear auth cookie
  → clear localforage
  → authClient.signOut()
  → remove better-auth localStorage keys
  → window.location.href = '/login' (hard navigation, destroys all JS state)
```

Hard navigation instead of Vue Router - see [rationale](/en/development/tech-decisions#hard-navigation-on-sign-out).

## Demo Mode

Demo mode works without authentication. Controlled by `finapp.isDemo` cookie.

- **Data:** 1000 random transactions, 18 categories, 6 wallets - generated in `useDemo.ts`
- **Storage:** localStorage/localforage only, no Convex calls
- **Store isolation:** all store mutation methods (`saveTrn`, `saveWallet`, etc.) check `isDemo` and skip Convex mutations + offline queue
- **Sign-out:** clears local data, no `authClient.signOut()` call

## Exchange Rates

- **Source:** OpenExchangeRates API (free tier)
- **Schedule:** Daily cron at 6:00 UTC (`convex/crons.ts`)
- **Storage:** `rates` table with `by_date` index (one row per day)
- **Frontend:** `useCurrenciesStore` fetches latest rates, triggers refresh if empty
- **Fallback:** If no rates exist in the database (new deployment), rates must be populated via the daily cron or manual `npx convex run`
- **Usage:** `getAmountInRate()` converts amounts to base currency for statistics

## Modal State

Modals use local `ref`s and `v-if` + `@close` emit pattern. Menu visibility is a module-level `ref` in `useMenuData.ts` (shared across callers). Other modals (e.g., wallet sort) use component-local `ref`s. No global modal registry - modals unmount automatically when parent components unmount (e.g., on sign-out, stores reset to `null` → layout condition becomes false → modals disappear).

## Sidebar Persistence

The desktop sidebar show/hide state is stored in a cookie (`finapp.isShowSidebar`), providing persistent state across page loads.

## Bottom Sheet

The mobile transaction form uses a custom `BottomSheet` component with `useBottomSheetDrag` - a hand-written drag implementation supporting touch and mouse. Features: configurable start-closing offset, direction detection, overlay opacity fade, scroll-aware behavior (ignores drag when content is scrolled), sort handle exclusion.

## Logging

`createLogger(prefix)` (`app/utils/logger.ts`) provides dev-only logging. In production, only `.error()` fires. All store operations and auth flows are instrumented with prefixed logs (`[wallets]`, `[trns]`, `[auth/middleware]`, etc.). Check the browser console during `pnpm dev`.

## PWA

- Strategy: `generateSW` from `@vite-pwa/nuxt` (no custom service worker)
- Precaching: all build assets are precached automatically
- `navigateFallback: '/'` serves SPA shell for all navigation (offline support)
- Runtime caching: Google Fonts, Iconify icons (CacheFirst)
- Start URL: `/dashboard`
- Manifest: `display: 'standalone'`

## Migration Tools

Convex backend includes internal mutations for data migration, callable via `npx convex run`:

| Function | Purpose |
|----------|---------|
| `migrate:deleteUserData` | Delete all data for a user |
| `migrate:insertWallets` | Bulk import wallets with ID remapping |
| `migrate:insertCategories` | Bulk import categories with parentId remapping |
| `migrate:insertTrnsBatch` | Bulk import transactions |
| `migrate:insertUserSettings` | Import user settings |

Migration script: `utils/migrate/firebase.js` - reads Firebase JSON export, transforms data, and calls the mutations above. See [Migration from Firebase](/en/development/migration) for details.
