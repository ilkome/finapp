Транзакции используют инкрементальную (дельта) синхронизацию, чтобы не загружать все данные при каждом запуске. Вызывается из `initTrns()` при [инициализации приложения](/ru/development/architecture#инициализация-приложения).

## Как это работает

1. Чтение `syncMeta` из LocalForage
2. **Нет syncMeta или кеша** → `fullSync()`: загрузка всего через пагинированный `trns.list`, сохранение syncMeta
3. **Есть syncMeta + кеш** → `deltaSync()`: загрузка только изменений через пагинированный `trns.delta` + `trns.idsHash` (параллельно)
4. При ошибке дельта или несовпадении хеша → откат на `fullSync()`
5. Оба пути заканчиваются `mergeOfflineOps()` + `setTrns()`

## Запросы Convex

| Запрос | Аргументы | Возвращает | Индекс |
|--------|-----------|------------|--------|
| `trns.list` | `paginationOpts` | Пагинированные транзакции (`PaginationResult`) | `by_user_date` |
| `trns.delta` | `paginationOpts`, `since: number` | Пагинированные транзакции с `updatedAt > since` (`PaginationResult`) | `by_user_updatedAt` |
| `trns.idsHash` | - | XOR-хеш всех ID транзакций (строка) | `by_user` |

## Метаданные синхронизации

Хранятся в LocalForage под ключом `finapp.trns.syncMeta`:

```typescript
type TrnsSyncMeta = {
  lastSyncedAt: number   // timestamp последней успешной синхронизации
  idsHash: string        // XOR-хеш всех ID транзакций при последней синхронизации
}
```

## Обнаружение удалений

Поскольку Convex использует жёсткое удаление (без поля `deletedAt`), удаления обнаруживаются сравнением хешей:

1. `deltaSync()` применяет дельта-изменения к кешированным данным
2. Вычисляет локальный XOR-хеш из всех не-локальных ID в результирующих данных
3. Сравнивает с `trns.idsHash()` с сервера
4. **Хеши совпадают** → удалений нет, синхронизация завершена
5. **Хеши не совпадают** → обнаружены удаления или пропущенные изменения → откат на `fullSync()`

Этот подход прост и надёжен. Удаления редки (1-2 в месяц), поэтому разовый `fullSync()` при несовпадении хешей - приемлемый компромисс.

## Триггеры полной синхронизации

Полная синхронизация запускается когда:
- Первый запуск приложения (нет syncMeta в LocalForage)
- Кеш LocalForage очищен
- Дельта-синхронизация не удалась (сетевая ошибка, повреждённые данные)
- Несовпадение хешей после применения дельта (обнаружены удаления)

## Взаимодействие с офлайн-очередью

Дельта-синхронизация полностью совместима с [офлайн-очередью](/ru/development/offline-queue):
- `mergeOfflineOps()` запускается после полной и дельта-синхронизации
- `cleanupFrontendIds()` удаляет устаревшие frontend ID из объединённых данных, которые больше не присутствуют в ожидающих операциях (уже синхронизированы в предыдущей сессии, но остались в кеше)
- Транзакции, созданные офлайн (frontend ID), не затрагиваются обнаружением удалений (сравниваются только Convex ID)
- Воспроизведение офлайн-очереди происходит после завершения `initTrns()`

## Производительность

| Метрика | Полная синхронизация (10К транзакций) | Дельта-синхронизация (10 изменений) |
|---------|--------------------------------------|-------------------------------------|
| Сеть | ~2 МБ (пагинированно) | ~2 КБ (дельта) + хеш |
| Парсинг | 10К объектов | 10 объектов |
| Сокращение | - | ~99% сети и парсинга |

## Ключевые файлы

| Файл | Роль |
|------|------|
| `convex/trns.ts` | Запросы `list`, `delta`, `idsHash` |
| `convex/schema.ts` | Индекс `by_user_updatedAt` |
| `app/components/trns/useTrnsStore.ts` | `fullSync()`, `deltaSync()`, `initTrns()`, `fetchAllPages()` |
| `app/composables/useStoreSync.ts` | `mergeOfflineOps()`, `cleanupFrontendIds()` |
| `app/components/offline/storageKeys.ts` | Ключ хранения `trnsSyncMeta` |
