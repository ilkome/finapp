Осознанные технические решения и их обоснование.

## `trns.list` / `trns.delta` - пагинация Convex

`convex/trns.ts` использует встроенный API `.paginate()` Convex. Фронтенд-хелпер `fetchAllPages()` проходит по всем страницам для сбора полного результата.

**Почему пагинация вместо JSON.stringify:**
Ранее запросы использовали `JSON.stringify` для обхода лимита Convex в 8 192 элемента массива. Пагинация - стандартный подход Convex: без накладных расходов на сериализацию, с правильной типизацией, нативная обработка больших наборов данных без обходных путей.

## Инкрементальная синхронизация (дельта)

`initTrns()` использует инкрементальную синхронизацию вместо полной загрузки при каждом запуске. Подробнее: [Инкрементальная синхронизация](/ru/development/sync).

**Почему не реактивные запросы Convex:**
Convex поддерживает подписки в реальном времени, но они требуют постоянного WebSocket-соединения и отправляют полный результат запроса при любом изменении. Для 10К+ транзакций это неэффективно. Дельта-синхронизация загружает только изменённые записи.

**Почему несовпадение хешей → fullSync вместо запроса ids:**
Ранее обнаружение удалений использовало отдельный запрос `trns.ids` для загрузки всех ID и поиска удалённых. Этот запрос был сломан при 8 000+ транзакциях (упирался в лимит 8 192 элемента массива). Новый подход: после применения дельта вычислить локальный XOR-хеш и сравнить с серверным. При несовпадении - откат на `fullSync()`. Удаления редки (1-2 в месяц), поэтому разовая полная синхронизация - приемлемый компромисс, проще и надёжнее поддержки пагинированного запроса ID.

**Почему не soft-delete:**
Soft-delete (поле `deletedAt`) потребовал бы миграции всех существующих данных, модификации всех запросов для фильтрации удалённых записей и периодической очистки. Обнаружение на основе хешей проще и не требует изменений схемы.

**Почему дельта-синхронизация только для транзакций:**
Категории и кошельки - маленькие наборы данных (десятки-сотни записей). Полная синхронизация достаточно быстра. Транзакции - единственная сущность, которая растёт до тысяч записей.

## Оптимистичный UI с fire-and-forget мутациями

Методы хранилища обновляют UI немедленно, затем запускают мутации Convex без ожидания. [Офлайн-очередь](/ru/development/offline-queue) фиксирует каждую операцию, поэтому ничего не теряется даже при ошибке мутации.

**Почему fire-and-forget:**
Ожидание мутаций блокировало бы UI и делало приложение медленным. Оптимистичные обновления дают мгновенную обратную связь. Механизм офлайн-очереди + воспроизведения обеспечивает итоговую согласованность.

**Почему `shallowRef` для элементов хранилища:**
Все Pinia-хранилища используют `shallowRef` вместо `ref` для карты элементов. Это позволяет избежать глубокого отслеживания реактивности на потенциально тысячах объектов, значительно улучшая производительность. Методы хранилища создают новые ссылки на объекты при мутации для запуска реактивности.

## Система Frontend ID

Новые сущности получают временные ID с префиксом `local_` (например, `local_a1b2c3`). После успешного создания на сервере ID переназначается на Convex ID. Полный жизненный цикл: [Офлайн-очередь - Система Frontend ID](/ru/development/offline-queue#система-frontend-id).

**Почему префикс `local_`:**
Явный префикс делает намерение понятным в коде - `isLocalId(id)` проверяет `id.startsWith('local_')`. Нет зависимости от внутреннего формата Convex ID.

**Почему не UUID:**
Короткий случайный суффикс достаточен для однопользовательского приложения. UUID длиннее и сложнее читать в логах.

## Жёсткая навигация при выходе

Выход из системы использует `window.location.href = '/login'` вместо навигации Vue Router.

**Почему:**
Жёсткая навигация уничтожает всё JavaScript-состояние - Pinia-хранилища, WebSocket-соединения, реактивные наблюдатели, кешированные данные. Это предотвращает утечку устаревшего состояния авторизации в следующую сессию. `navigateTo()` Vue Router сохранил бы всё состояние в памяти.

## Безопасный арифметический парсер в калькуляторе

Калькулятор формы транзакции использует рекурсивный нисходящий парсер (`app/components/trnForm/utils/calculate.ts`) вместо `eval()` или `new Function()`.

**Почему:**
`new Function(expression)` - вектор инъекции кода: вредоносная строка может выполнить произвольный JavaScript. Также требует `unsafe-eval` в Content-Security-Policy. Кастомный парсер поддерживает только `+`, `-`, `*`, `/` над числовыми литералами, отклоняя всё остальное.

## Каскадное удаление в Convex

Convex не имеет ограничений внешних ключей, поэтому каскадное удаление выполняется вручную - `wallets.remove` и `categories.remove` удаляют все связанные транзакции в одной мутации. Фронтенд отправляет только удаление родительской сущности в [офлайн-очередь](/ru/development/offline-queue#каскадное-удаление-на-бэкенде).

**Почему пагинированными батчами (500 элементов):**
Convex ограничивает количество записанных документов за транзакцию до 16 000. Кошелёк с тысячами транзакций может превысить этот лимит в одной мутации. Каскадное удаление теперь пагинируется: получить батч из 500 ID транзакций, удалить их, повторить пока не останется ни одной, затем удалить родительскую сущность.

## Вычисляемые дочерние категории вместо денормализованных `childIds`

Категории хранят только `parentId`. Дочерние ID вычисляются динамически фильтрацией категорий по `parentId === id`.

**Почему:**
Ранее родительские категории хранили массив `childIds`. Это требовало синхронизации обеих сторон при каждом перемещении, создании или удалении - источник багов. Вычисление детей из `parentId` - единый источник правды без проблем синхронизации. Количество категорий достаточно мало (десятки-сотни), чтобы стоимость поиска была ничтожной.

## Ограничение вложенности категорий до 2 уровней

Категории поддерживают только родитель → дочерний, без более глубокой вложенности.

**Почему:**
Более глубокая вложенность усложняет UI (разбивка статистики, выбор категорий, хлебные крошки) без практической пользы для учёта личных финансов. Два уровня покрывают 99% случаев использования (например, Еда > Продукты).

## Приоритет инъекции темы

Стили темы используют `tagPriority: -2` в `useHead()`, а скрипты плагинов используют `tagPriority: -1`.

**Почему:**
Это гарантирует применение CSS-переменных темы до первой отрисовки. Цепочка приоритетов: `useHead` стили (-2) → скрипты плагинов (-1) → `@layer theme` (всегда проигрывает non-layered). Non-layered CSS в `theme.css` переопределил бы `useHead()` порядком в документе, поэтому `--ui-radius` и `--ui-primary` не должны устанавливаться там.

## Лимиты Convex

Справка ([документация](https://docs.convex.dev/production/state/limits)):
- Возвращаемое значение функции: 16 МиБ
- Чтение данных за транзакцию: 16 МиБ
- Просканированных документов за запрос: 32 000
- Размер документа: 1 МиБ
- Элементов массива в документе: 8 192
- Записанных документов за транзакцию: 16 000
- Время выполнения запроса: 1 сек (пользовательский код, исключая операции БД)
