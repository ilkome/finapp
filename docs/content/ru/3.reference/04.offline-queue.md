Единая упорядоченная офлайн-очередь со свёрткой операций. Все ожидающие операции (создание/обновление/удаление) для кошельков, категорий и транзакций хранятся в одном упорядоченном массиве в localforage под ключом `finapp.offline.queue`.

## Архитектура

### Ключевые файлы

| Файл | Описание |
|------|----------|
| `app/components/offline/types.ts` | Типы: `OfflineOp`, `EntityType` |
| `app/components/offline/helpers.ts` | API очереди: push, remove, get, clear |
| `app/components/offline/replay.ts` | Воспроизведение очереди при перезапуске |
| `app/components/offline/storageKeys.ts` | Ключи LocalForage для кешей сущностей |

### Правила свёртки

При добавлении новой операции очередь находит последнюю операцию с тем же `entity + id` и выполняет свёртку:

| Существующая | Новая | Результат |
|-------------|-------|-----------|
| create | update | create (объединённые данные) |
| create | delete | удаление обеих (не достигли сервера) |
| update | update | update (объединённые данные) |
| update | delete | замена на delete |
| delete | * | игнорирование новой (delete - терминальная) |
| - | * | добавление как есть |

### Сериализация

`pushOfflineOp` использует цепочку промисов для сериализации параллельных записей - `await` не требуется в местах вызова.

### Каскадное удаление на бэкенде

`wallets.remove` и `categories.remove` на бэкенде Convex каскадно удаляют все связанные транзакции. Фронтенд НЕ добавляет отдельные операции удаления транзакций при удалении кошелька или категории - в очередь попадает только удаление родительской сущности. `removeTrnsFromStore()` обрабатывает оптимистичную очистку UI без мутаций.

## Система Frontend ID

Новые сущности нуждаются в ID до ответа сервера. Фронтенд генерирует временные ID, которые затем переназначаются на реальные Convex ID.

### Генерация

`generateId()` (`utils/generateId.ts`) создаёт ID с префиксом `local_` + 6 случайных base-36 символов. Пример: `local_a1b2c3`.

### Отличие от Convex ID

`isLocalId(id)` (`app/utils/convexId.ts`) проверяет `id.startsWith('local_')`. Любой ID с префиксом `local_` - frontend ID.

Эта проверка используется по всей кодовой базе:
- Хранилища: определение создания vs обновления (`!isLocalId(id) && id in items`)
- Дельта-синхронизация: исключение локальных ID из обнаружения удалений
- Пакетное удаление: фильтрация локальных ID для серверной мутации
- Воспроизведение: пропуск операций удаления для локальных ID (нечего удалять на сервере)

### Переназначение ID

После успешной мутации создания сервер возвращает новый Convex ID. `handleMutationResult()` (`app/composables/useStoreSync.ts`) заменяет ключ frontend ID в карте хранилища:

```
items[frontendId] → items[convexId]  (те же данные, новый ключ)
```

Переназначенное хранилище немедленно сохраняется в localforage.

### Очистка устаревших ID

`cleanupFrontendIds()` (`app/utils/convexId.ts`) запускается в `mergeOfflineOps()` при каждой инициализации. Удаляет локальные ID из кешированных данных, которых больше нет в ожидающей офлайн-очереди - это ID из предыдущей сессии, которые были либо успешно переназначены, либо брошены.

### Где создаются Frontend ID

| Сущность | Файл | Код |
|----------|------|-----|
| Транзакция | `app/components/trnForm/useTrnsFormStore.ts` | `id: values.trnId ?? generateId()` |
| Категория | `app/components/categories/Form.vue` | `editCategoryId = props.categoryId ?? generateId()` |
| Кошелёк | `app/components/wallets/Form.vue` | `editWalletId = props.walletId ?? generateId()` |

### Крайние случаи

**Параллельные создания:** Каждое переназначение независимо - деструктуризация извлекает конкретный frontend ID и пересобирает объект. Конфликтов между параллельными мутациями создания нет.

**Краш до переназначения:** Если приложение падает после успешной мутации, но до выполнения `.then()`, `cleanupFrontendIds()` следующей сессии удалит устаревший frontend ID из кеша, а сущность существует на сервере с Convex ID.

## Сценарии

### Создание

#### Онлайн
1. Пользователь создаёт сущность (кошелёк/категорию/транзакцию)
2. Хранилище генерирует frontend ID (например, `local_abc123`)
3. Оптимистичный UI: сущность появляется в хранилище немедленно
4. `pushOfflineOp({ type: 'create', ... })` сохраняет в очередь
5. Мутация Convex запускается (fire-and-forget)
6. `.then()`: удаление операции из очереди, переназначение frontend ID на Convex ID в хранилище

#### Офлайн
1. Шаги 1-4 такие же, как онлайн
2. Мутация не удаётся (сетевая ошибка) - `.catch()` показывает toast
3. Операция остаётся в очереди
4. Сущность видна в UI с frontend ID

#### Перезапуск после офлайн-создания
1. `initTrns` получает серверные данные (полная синхронизация при первой загрузке, [дельта-синхронизация](/ru/development/sync) при последующих); `initWallets/initCategories` получают полные данные
2. Ожидающие операции из очереди объединяются поверх (записи с frontend ID добавляются в хранилище)
3. `replayOfflineQueue()` запускается:
   - Frontend ID + create/update → отправка в метод хранилища → мутация запускается → `.then()` переназначает ID
   - Frontend ID + delete → пропускается (нечего удалять на сервере)
4. Сущность получает настоящий Convex ID

### Обновление

#### Онлайн
1. Пользователь редактирует сущность
2. `isExisting = !isLocalId(id) && id in items` → `true`
3. Оптимистичный UI: хранилище обновляется немедленно
4. `pushOfflineOp({ type: 'update', ... })` - свёртка с существующими операциями
5. Мутация Convex `update` запускается
6. `.then()`: удаление операции из очереди

#### Офлайн
1. Шаги 1-4 такие же, как онлайн
2. Мутация не удаётся - операция остаётся в очереди
3. Несколько редактирований сворачиваются: update + update → один update с объединёнными данными

#### Перезапуск после офлайн-обновления
1. Инициализация получает серверные данные - старую версию (через [дельта-синхронизацию](/ru/development/sync) или полную синхронизацию)
2. Ожидающее обновление объединяется поверх → UI показывает последнюю версию
3. Воспроизведение отправляет update → мутация запускается → синхронизировано

### Удаление

#### Онлайн
1. Пользователь удаляет сущность
2. Оптимистичный UI: удаление из хранилища немедленно
3. Для кошельков/категорий: `removeTrnsFromStore()` удаляет дочерние транзакции из UI
4. `pushOfflineOp({ type: 'delete', ... })` - свёртка с существующими операциями
5. Мутация Convex `remove` запускается (каскадное удаление транзакций на бэкенде)
6. `.then()`: удаление операции из очереди

#### Офлайн
1. Шаги 1-4 такие же, как онлайн
2. Мутация не удаётся - операция остаётся в очереди
3. Свёртка: если сущность была создана офлайн и затем удалена → обе операции удаляются (не достигли сервера)

#### Перезапуск после офлайн-удаления
1. Инициализация получает серверные данные - сущность всё ещё существует на сервере (через [дельта-синхронизацию](/ru/development/sync) или полную синхронизацию)
2. Ожидающее удаление убирает сущность из объединённых данных
3. Воспроизведение отправляет delete → мутация запускается → синхронизировано

### Крайние случаи

#### Создание + редактирование офлайн, затем перезапуск
1. Создание: `pushOfflineOp({ type: 'create', data: v1 })`
2. Редактирование: свёртка до `create` с объединёнными данными (v1 + v2)
3. Перезапуск: одна операция create воспроизводится → сервер получает финальную версию

#### Создание + удаление офлайн, затем перезапуск
1. Создание: `pushOfflineOp({ type: 'create', ... })`
2. Удаление: свёртка → обе операции удаляются из очереди
3. Перезапуск: нечего воспроизводить, сущность никогда не существовала на сервере

#### Удаление кошелька/категории с транзакциями офлайн
1. `pushOfflineOp({ type: 'delete', entity: 'wallets', id })` - только удаление кошелька в очереди
2. `removeTrnsFromStore(trnsIds)` - оптимистичная очистка UI, без операций очереди для транзакций
3. Перезапуск: удаление кошелька воспроизводится → бэкенд каскадно удаляет транзакции

#### Осиротевшие транзакции при воспроизведении
1. Транзакция создана офлайн со ссылкой на кошелёк/категорию
2. Этот кошелёк/категория удалён (тоже офлайн или с другого устройства)
3. Воспроизведение обнаруживает сироту: кошелёк/категория больше нет в хранилище
4. Операция сиротской транзакции удаляется из очереди, пользователю показывается toast

#### Параллельные вызовы pushOfflineOp
Сериализация цепочкой промисов гарантирует, что записи не перезаписывают друг друга:
```typescript
let writeChain = Promise.resolve()
export function pushOfflineOp(op) {
  writeChain = writeChain.then(() => impl(op), () => impl(op))
  return writeChain
}
```

#### Флаг isReplaying
Во время воспроизведения методы хранилищ пропускают `pushOfflineOp`, чтобы не добавлять в очередь операции, которые уже воспроизводятся. Флаг устанавливается перед циклом воспроизведения и сбрасывается в `finally`.

## Известные ограничения

### Порядок кошельков не сохраняется в offline queue

`saveWalletsOrder` использует fire-and-forget без offline queue. Если пользователь изменит порядок кошельков в offline и закроет приложение до успешной мутации - порядок будет потерян при следующей загрузке (сервер вернёт старый порядок, pending операции для merge нет).

Это сделано осознанно - сортировка кошельков выполняется редко, а добавление в очередь потребовало бы специального `id: 'order'`, отдельной логики merge в `initWallets` и дополнительной ветки в replay. Эта сложность не оправдана.
