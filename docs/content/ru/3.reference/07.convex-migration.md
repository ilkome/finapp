Сравнение конечного состояния приложения на `main` (v6, Firebase) и `convex` (v7, Convex).

## Бэкенд

| | main (Firebase) | convex |
|---|---|---|
| **База данных** | Firebase Realtime Database | Convex (backend-as-code) |
| **Авторизация** | Firebase Auth через `nuxt-vuefire` | Better Auth + `@convex-dev/better-auth` |
| **Доступ к данным** | Real-time listeners (`getDataAndWatch`) | Императивный `client.query()` / `client.mutation()` |
| **Валидация** | Firebase security rules (JSON) | Convex-валидаторы (`v.*`) + Zod-схемы |
| **Серверный код** | Нет (клиентский SDK) | Директория `convex/`: схема, мутации, запросы, HTTP, кроны |
| **Курсы валют** | Клиентский запрос | Серверный крон (ежедневно в 6:00 UTC) + таблица `rates` |
| **Настройки пользователя** | Только localStorage | Convex-таблица `userSettings` (baseCurrency, locale) с офлайн-очередью |
| **Индексы схемы** | N/A | Составные индексы: `by_user_date`, `by_user_wallet`, `by_user_category` для эффективных запросов |
| **Миграция** | N/A | `utils/migrate/firebase.js` &mdash; CLI для импорта JSON-экспортов Firebase в Convex |
| **Удаление аккаунта** | N/A | `deleteUserDataFull()` &mdash; пагинированная очистка таблиц приложения + авторизации |

### Что даёт Convex

- **Типизированные мутации** &mdash; полный TypeScript на обеих сторонах, автогенерация API-типов
- **Серверная валидация** &mdash; лимиты длины строк, диапазоны чисел, непустота полей, бизнес-правила (см. [Стратегия валидации](/ru/development/validation-strategy))
- **Один WebSocket** для всех операций (запросы, мутации, авторизация)
- **Пагинация** через `.paginate()` API для больших наборов данных (транзакции)
- **Дельта-синхронизация** &mdash; после начальной загрузки запрашиваются только изменённые записи (см. [Синхронизация](/ru/development/sync))

## Паттерн сторов

| | main | convex |
|---|---|---|
| **Источник данных** | Firebase listener &rarr; `setTrns()` | Convex query &rarr; `mergeOfflineOps()` &rarr; `set()` |
| **Persist** | `deepUnref()` + немедленный `localforage.setItem()` | Debounced 300ms `localforage.setItem()` (без глубокого клонирования) |
| **Офлайн-операции** | Ручные хелперы в `trns/helpers.ts`, произвольные ключи localStorage | Явная очередь: `pushSaveOp()`, `pushDeleteOp()`, `replay()` |
| **Система ID** | Firebase push ID | Префикс `local_` &rarr; замена на Convex ID после мутации |
| **Оптимистичный UI** | Частичный (fire-and-forget) | Полный: обновить UI &rarr; push op &rarr; fire mutation &rarr; remap ID при успехе |
| **Верификация синхронизации** | Нет | XOR-хеш (`fnv1a`) ID транзакций; сравнение хешей клиент &harr; сервер при каждой синхронизации |

Зависимость `vue-deepunref` удалена. Данные сторов используют `shallowRef`, поэтому глубокое клонирование при каждом persist было лишним.

## Производительность

### Алгоритмические улучшения

| Что | main | convex | Эффект |
|-----|------|--------|--------|
| **Итоги кошельков** | `getWalletTotal(id)` вызывается для каждого кошелька в `reduce` &mdash; каждый вызов фильтрует все транзакции: O(W&times;N) | `getWalletsTotals()` один проход, распределение через `Map`: O(N), затем O(1) для каждого кошелька | 5 кошельков &times; 10K трн: ~50K &rarr; ~10K операций |
| **Разбивка по интервалам** | Инлайн в `Item.vue`: `intervalsInRange.reduce()` фильтрует все транзакции на каждый интервал: O(N&times;I) | Извлечено в `bucketTrnsByIntervals()`: один проход + бинарный поиск: O(N&nbsp;log&nbsp;I) | Год/365 дней/5K трн: 1.8M &rarr; ~43K сравнений |
| **Фильтрация транзакций** | 6 последовательных `.filter()` через `reduce` + `walletsIds.some(w => includes(w))` O(W&sup2;) + сортировка всегда O(N&nbsp;log&nbsp;N) | Один `.filter()` с ранним выходом + `Set.has()` O(1) + сортировка опциональна | 5 промежуточных массивов + O(W&sup2;) &rarr; один проход + O(1); сортировка O(N&nbsp;log&nbsp;N) пропускается когда не нужна |
| **Поиск кошелька в getTotal** | `walletsIds.includes()`: O(W) на каждый перевод | `new Set(walletsIds).has()`: O(1) на каждый перевод | Линейный &rarr; константный на каждый вызов |
| **Генерация демо-данных** | `{ ...acc }` spread в `.reduce()` + `getTransactibleIds()` вызывается дважды на итерацию | Прямое `acc[i] = ...` + кеширование перед циклом | O(N&sup2;) &rarr; O(N) |
| **Статистика категорий** | `categoriesStat` жадно вычисляет и `grouped`, и `ungrouped` на каждое изменение; `filter().at(0)` для наибольшей | Ленивый `computed()` на каждый вариант, выполняется только при обращении; `.find()` останавливается на первом совпадении | До 2&times; меньше вызовов `computeCategoriesWithData` |
| **Фильтр вертикальных категорий** | `verticalCategories.filter(c => c.value !== 0)` вызывается дважды в шаблоне (заголовок + v-for): пересчитывается на каждый рендер | Кеширован в `visibleVerticalCategories` computed: фильтрация один раз, переиспользование | 2 прохода фильтра &rarr; 1 кешированный |
| **Форматирование чисел** | Библиотека `currency.js` + `currencies.find()` O(N) дважды на вызов | Нативный `Intl.NumberFormat` с кешем в `Map` + `currencyMap.get()` O(1) | Нет зависимости; форматтер создаётся один раз на precision |
| **Последняя транзакция** | `Object.keys().sort().find()` &mdash; сортирует все транзакции O(N&nbsp;log&nbsp;N) затем ищет | Один проход O(N), отслеживая максимальную дату | 10K трн: ~130K ops &rarr; ~10K ops |
| **Проверка родителя категории** | `categoriesForBeParent` вызывает `Object.values(trns).some()` O(N) на каждую корневую категорию: O(M&times;N) | Предготовленный `usedCategoryIds` Set через `for...in` O(N) + `Set.has()` O(1) на проверку; без промежуточных массивов | 200 категорий &times; 10K трн: ~2M &rarr; ~10K ops |
| **Недавние категории** | `Object.keys(trns).filter().sort()` O(N&nbsp;log&nbsp;N) + `acc.some(c => c.id)` O(K&sup2;) дедупликация + `includes()` O(W) на категорию | Один проход O(N) через `Map` с последней датой + `entries().toSorted()` O(K&nbsp;log&nbsp;K) + `Set.has()` O(1) | 10K трн: ~130K + K&sup2; &rarr; ~10K + K&nbsp;log&nbsp;K ops |
| **Транзактируемые категории** | `.reduce()` с `acc.includes()` O(M) + `childIds.filter(!acc.includes)` O(C&times;M): O(M&sup2;) | `Map<parentId, childIds[]>` за O(M) + `Set` дедупликация O(1) на проверку | 200 категорий: ~40K &rarr; ~200 ops |
| **Определение переводов** | `transferCategoriesIds.includes(categoryId)` O(W) линейный поиск на каждую транзакцию | `trn.type === TrnType.Transfer` O(1) сравнение enum | Линейный &rarr; константный на проверку |
| **Подсчёт фильтров типов** | 3 отдельных `.filter()` прохода по trnsIds (расход, доход, перевод): O(3N) | Один цикл с объектом счётчиков: O(N) | 3 полных прохода &rarr; 1 проход |
| **Кеш элементов в списке** | `computeTrnItem(id)` вызывается дважды на элемент в шаблоне (элемент + дата): O(2P) вызовов | Предвычисленная `trnItemsMap` через `Map`: O(P) вычислений + O(1) обращения | 30 элементов: ~60 &rarr; ~30 вычислений на рендер |
| **Кеш итогов групп** | `getTotalOfTrnsIds()` вызывается дважды на группу в шаблоне (доход + расход): O(2G) вызовов | Предвычисленная `groupTotals` Map + `paginatedTotal` computed: O(G) вычислений + O(1) обращения | 15 групп: ~30 &rarr; ~15 вычислений итогов на рендер |
| **Сортировка при группировке** | `categories.sort()` вызывается внутри `reduce` на каждый `.push()` дочерней категории: O(K&sup2;&nbsp;log&nbsp;K) на родителя | Сортировка один раз после цикла, только если `length > 1`: O(K&nbsp;log&nbsp;K) на родителя | 5 детей: ~50 &rarr; ~12 сравнений |
| **Итоги при группировке** | `getTotalOfTrnsIds(cat.trnsIds).sum` вызывается дважды на дочернюю категорию (для значения + суммы родителя) | Вычисляется один раз в `catTotal`, переиспользуется: один вызов на дочернюю | 2&times; меньше вызовов `getTotal` на дочернюю категорию |
| **Синхронизация данных** | Firebase: полная замена данных на каждое обновление: O(N) | Дельта-синхронизация: запрашиваются только изменённые записи &mdash; O(D) где D &laquo; N | Не обрабатывает неизменённые транзакции |
| **Генерация интервалов** | `list.unshift(current)` в цикле while: O(n) на вызов, O(n&sup2;) всего | `list.push(current)` + `list.reverse()`: O(1) на вызов, O(n) всего | Год/365 дней: ~66K сдвигов &rarr; ~365 push + 1 reverse |
| **Ключи Duration** | Динамический `{ [\`${period}s\`]: value }` &mdash; TS видит `Record<string, number>`, нет проверки типа `Duration` | `toDuration(period, value)` возвращает типизированный `Duration` через exhaustive switch | Безопасность на этапе компиляции; новые варианты периодов вызывают ошибки, а не молчаливый fallback |
| **Форматирование дат** | Каждая format-функция вызывается дважды (`type: 'start'` + `type: 'end'`), результаты конкатенируются; 3 IIFE switch-блока для сравнения периодов | Каждая format-функция возвращает полную строку; `isSamePeriod(a, b, period)` заменяет все 3 switch-блока | ~40% меньше строк в `useGetDateRange`; без двойных вызовов |
| **Вычисление средних** | `sum !== 0` проверяется 3 раза + `Object.keys(items).length > 0` гард | Ранний выход при `sum === 0`; среднее за день всегда присутствует (диапазон &ge; 2 гарантирован) | 1 проверка вместо 4 |
| **Сортировка избранных категорий** | Инлайн-сортировка дублирует логику parent-name + name (8 строк) | Переиспользует утилиту `compareCategoriesByParentAndName` (1 строка) | DRY; одна реализация сортировки |
| **Цикл недавних категорий** | `.reduce()` проходит все записи даже после достижения лимита `maxCategories` | `for` цикл с `break` останавливается на лимите | 200 категорий, лимит 16: ~200 &rarr; ~16 итераций |
| **Итерация недавних категорий** | `for...of Object.keys(trnsItems)` создает промежуточный массив | `for...in trnsItems` итерирует напрямую по объекту | Без аллокации промежуточного массива для 10K+ транзакций |
| **Переиспользование ID категорий** | `Object.keys(items.value)` вызывается отдельно в `categoriesRootIds`, `favoriteCategoriesIds` | Оба переиспользуют computed `categoriesIds` | Ключи вычисляются один раз, используются всеми зависимыми |
| **Переиспользование transactible IDs** | `categoriesIdsForTrnValues` повторно фильтрует все категории с проверкой `!hasChildren` | Делегирует уже вычисленному `transactibleIds` + `.filter(id !== 'transfer')` | Без повторного O(M) сканирования родителей |
| **Дедупликация children IDs** | `getChildrenIdsOrParent` дублирует логику `.filter()` из `getChildrenIds` | Делегирует `getChildrenIds()`, проверяет `.length` | Одна реализация фильтра |

## Безопасность

| Что | main | convex |
|-----|------|--------|
| **Парсер калькулятора** | `new Function(expression)` &mdash; риск инъекции кода, требует `unsafe-eval` CSP | Рекурсивный парсер &mdash; только `+`,`-`,`*`,`/` над числами |
| **Валидация ввода** | Firebase rules (клиент не может проверить) | Convex-валидаторы: лимиты строк, диапазоны чисел, проверки полей |

| **Каскадные удаления** | N/A (Firebase) | Пагинированные батчи по 500 (лимит Convex 16K записей) |
| **Целостность данных** | Не проверяется в коде | Transfer &harr; `categoryId='transfer'`, нормализация при смене типа, уникальность имён |
| **Защита редиректов** | Без валидации | `getSafeRedirectPath()` &mdash; только относительные пути, начинающиеся с `/` (не `//`) |

## TypeScript

| | main | convex |
|---|---|---|
| **Strict mode** | Включён, но есть неразрешённые ошибки | Ноль ошибок TypeScript |
| **Касты типов** | `as any` по кодовой базе | Ноль `as any` &mdash; type guards, `@vue-ignore`, узкие касты |
| **Определение переводов** | `transferCategoriesIds` (computed из данных категорий) | `TrnType.Transfer` enum + discriminated union типы |
| **Валидация на рантайме** | Нет | Zod-схемы для настроек, курсов, конфига статистики, параметров дат |
| **Функции дат** | `getStartOf(date, string)`, `getEndOf(date, string)` &mdash; `default` fallback на day | Типизированы как `Period`, exhaustive `switch` с `case 'day'` &mdash; новые варианты вызывают ошибку компиляции |
| **Конструкция Duration** | Динамический `{ [\`${period}s\`]: value }` &mdash; TS видит `Record<string, number>` | `toDuration(period, value): Duration` &mdash; типизированный возврат через exhaustive switch |
| **Маппинг типов статистики** | `Record<string, TrnType[]>` с `??` fallback | `Record<SeriesSlugSelected \| StatTabSlug, TrnType[]>` &mdash; все ключи покрыты, fallback не нужен |
| **Ключ средних** | `key as keyof TotalReturns` небезопасный каст + `as number` | Типизированный `key: keyof TotalReturns` через явный маппинг `'netIncome'` &rarr; `'sum'`, без кастов |
| **Fallback курсов** | `rates[code] \|\| 1`, `wallet?.currency \|\| 'USD'` &mdash; falsy `0`/`''` триггерит fallback | `rates[code] ?? 1`, `wallet?.currency ?? 'USD'` &mdash; только `null`/`undefined` триггерит fallback |
| **Парсинг query-параметров** | `if (data.rangeOffset)` &mdash; falsy `0` пропускается | `if (data.x !== undefined)` &mdash; нулевые значения применяются корректно |
| **Пропсы total** | `baseCurrencyCode?: string` нестрогий тип | `baseCurrencyCode?: CurrencyCode` &mdash; согласованно с `getAmountInRate` |
| **Параметры стат. категорий** | `trnsItems: Record<TrnId, { categoryId?: string }>` инлайн-тип | `trnsItems: Record<TrnId, Pick<TrnItem, 'categoryId'>>` &mdash; привязан к исходному типу |
| **Null guard в сортировке** | `sortCategoriesByAmount` проверяет `if (!a \|\| !b)` в рантайме | Гард удален &mdash; TS уже гарантирует параметры `CategoryWithData`; явный возвращаемый тип `: number` |
