## Инициализация приложения

```
запуск приложения
├── Плагины
│   ├── theme.ts (enforce: 'post')
│   │   └── Чтение темы из localStorage → обновление appConfig
│   └── convex.ts
│       └── Создание WebSocket-клиента → настройка авторизации
├── app.vue
│   ├── useHead() - CSS-переменные темы (tagPriority: -2)
│   ├── useGuard() - логика редиректа авторизации
│   └── рендер layout + page
└── useInitApp (на авторизованной странице)
    ├── loadDataFromCache() - параллельное чтение из localforage (6 элементов)
    ├── loadDataFromDB()
    │   ├── await waitForConvexAuth() - ожидание аутентификации WebSocket
    │   ├── параллельная инициализация хранилищ (5 хранилищ)
    │   │   ├── initCurrencies()
    │   │   ├── initUserSettings()
    │   │   ├── initWallets()
    │   │   ├── initCategories()
    │   │   └── initTrns() - дельта или полная синхронизация
    │   └── replayOfflineQueue() - воспроизведение ожидающих офлайн-операций
```

## Структура проекта

```
app/           # Nuxt: компоненты, composables, страницы, плагины
convex/        # Бэкенд: схема, queries, mutations, auth, Telegram-бот
services/      # Клиентская инфраструктура (фабрика Convex-клиента, конвертеры API)
utils/         # Чистые хелперы (id, logger, типы)
utils/migrate/ # Скрипты миграции Firebase → Convex (data.json в gitignore)
i18n/locales/  # en-US.js, ru-RU.js
```

Неочевидные соглашения:
- `convex/_generated/` создаётся Convex CLI - не редактировать.
- `services/convex/` вынесен из `convex/` намеренно: `convex/` - это деплоящийся бэкенд, `services/` - только фронтенд-прослойка (фабрика клиента, конвертеры doc → map).
- Каждая фича в `app/components/<feature>/` содержит свой Pinia store, форму, список и типы - отдельной папки `stores/` нет.

## Паттерн хранилища

Все Pinia-хранилища (`useTrnsStore`, `useWalletsStore`, `useCategoriesStore`) следуют одному паттерну:

```
items: shallowRef          # реактивное состояние (shallow для производительности)
init()                     # запрос из Convex → mergeOfflineOps → set
set(data)                  # items.value = data + debouncedPersist (300мс)
save({ id, values })       # оптимистичный UI → pushSaveOp → fire mutation
delete(id)                 # оптимистичный UI → pushDeleteOp → fire mutation
```

### Поток данных (save)

```
действие пользователя
  → save({ id, values })
    → isExisting = !isLocalId(id) && id in items
    → оптимистично: set({ ...items, [id]: values })
    → pushSaveOp({ entity, id, isDemo, isExisting, values })
    → fire mutation (без await)
      → .then(): removeOfflineOp + переназначение frontend ID
      → .catch(): показать toast с ошибкой
```

### Поток данных (init)

```
init()
  → запрос из Convex (query)
  → конвертация в map (удаление _id, _creationTime, userId)
  → mergeOfflineOps(data, entity)
    → чтение ожидающих операций из очереди
    → применение create/update операций поверх серверных данных
    → удаление устаревших frontend ID
  → set(data)
```

## Доступ к клиенту Convex

| Composable | Возвращает | Назначение |
|------------|-----------|------------|
| `useConvexClient()` | `ConvexClient` | Синглтон WebSocket-клиента |
| `useConvexClientWithApi()` | `{ api, client }` | Клиент + автогенерируемый объект `api` для типизированных запросов/мутаций |

Хранилища используют императивный `client.query()` и `client.mutation()`, так как управляют собственным кешированием и офлайн-слиянием.

**Поток авторизации (клиент - `plugins/convex.ts`):**
1. При инициализации плагина: если `hasAuthCookie()` && онлайн → вызов `client.setAuth(fetchToken)` для параллельного получения токена с разрешением сессии
2. `fetchToken` использует `authClient.convex.token()` (Better Auth `convexClient()` плагин) - `crossDomainClient` автоматически пробрасывает cookie через `localStorage['better-auth_cookie']`
3. `authReadyPromise` разрешается после первого получения токена (успех или ошибка) - `loadDataFromDB()` ожидает это перед запросами к Convex
4. Watch сессии действует только на определённые результаты (пользователь найден → `setAuth()`, пользователь отсутствует без ошибки → `clearAuth()`). Сетевые ошибки и `isPending` игнорируются для предотвращения выхода при офлайне
5. Событие `online`: пересоздаёт `authReadyPromise` и вызывает `setAuth()` для нового токена
6. `ensureConvexAuth()`: вызывается из `/auth/callback` для установки авторизации ДО SPA-навигации на dashboard

## Система Frontend ID

Новые сущности получают временные ID с префиксом `local_` (например, `local_a1b2c3`). `isLocalId(id)` проверяет `id.startsWith('local_')` для их различения. См. [обоснование](/ru/development/tech-decisions#система-frontend-id).

Frontend ID переназначаются на Convex ID в `handleMutationResult()` после успешных мутаций создания. Хранилище заменяет `items[frontendId]` на `items[convexId]`.

## Защита авторизации

### Слои авторизации

Авторизация использует два слоя:

1. **Route middleware** (`auth.global.ts`): проверяет cookie `finapp.localAuthUid` и решает, разрешить навигацию или редиректить на `/login`. При отсутствии сети пропускает сетевые запросы, доверяя кешированному cookie.
2. **Клиентская защита** (`useGuard()` в `app.vue`): асинхронно следит за сессией Better Auth и обрабатывает истечение сессии после начальной загрузки. При неожиданной потере сессии (не через sign-out) cookie `finapp.localAuthUid` очищается.

### OTT-авторизация

Google OAuth использует одноразовый токен (OTT) для кросс-доменной авторизации:

1. Google OAuth перенаправляет обратно с параметром `?ott=...`
2. Глобальный middleware перехватывает любой маршрут с `ott` в запросе и перенаправляет на `/auth/callback`
3. Страница callback верифицирует OTT с бэкендом Convex Better Auth
4. При успехе: устанавливает cookie авторизации, вызывает `ensureConvexAuth()` для немедленного старта авторизации Convex, затем навигация через `navigateTo()` на исходный URL

### Защита от редиректов

`getSafeRedirectPath()` (`app/utils/redirect.ts`) валидирует параметры `?redirect=` - разрешены только относительные пути, начинающиеся с `/` (не `//`), что предотвращает атаки открытого перенаправления.

### Логика защиты

`useGuard()` в `app.vue` наблюдает за `currentUser`:
- Авторизован + на `/login` → редирект на `/dashboard` (или путь из `?redirect=`)
- Не авторизован + не на `/login` → редирект на `/login`
- Флаг `isSigningOut` предотвращает цикл редиректов при выходе

## Офлайн / Онлайн

Приложение сохраняет состояние авторизации и данные при офлайне:

- **Cookie авторизации сохраняется офлайн:** Watch сессии игнорирует сетевые ошибки (`s.error`) - только определённые ответы «нет пользователя» очищают cookie. Предотвращает выход при офлайне или нестабильном соединении.
- **Получение токена пропускается офлайн:** `fetchToken` проверяет `!navigator.onLine` перед HTTP-запросами - избегает шумных ошибок в консоли.
- **Переподключение:** Слушатель события `online` пересоздаёт `authReadyPromise` и вызывает `setAuth()`, чтобы `loadDataFromDB()` (который ожидает `waitForConvexAuth()`) получил свежий токен.
- **Route middleware:** `auth.global.ts` проверяет `hasAuthCookie()` синхронно - без сетевых запросов, работает офлайн.
- **PWA:** Service worker прекеширует все ассеты сборки + `NavigationRoute` отдаёт SPA-оболочку для всех навигаций.

## Поток выхода

```
signOut()
  → isSigningOut = true (предотвращение цикла редиректов)
  → очистка данных хранилищ (trns, categories, wallets, user)
  → очистка cookie авторизации
  → очистка localforage
  → authClient.signOut()
  → удаление ключей better-auth из localStorage
  → window.location.href = '/login' (жёсткая навигация, уничтожение всего JS-состояния)
```

Жёсткая навигация вместо Vue Router - см. [обоснование](/ru/development/tech-decisions#жёсткая-навигация-при-выходе).

## Демо режим

Демо режим работает без авторизации. Управляется cookie `finapp.isDemo`.

- **Данные:** 1000 случайных транзакций, 18 категорий, 6 кошельков - генерируются в `useDemo.ts`
- **Хранение:** только localStorage/localforage, без запросов к Convex
- **Изоляция хранилищ:** все методы мутации (`saveTrn`, `saveWallet` и т.д.) проверяют `isDemo` и пропускают мутации Convex + офлайн-очередь
- **Выход:** очистка локальных данных, без вызова `authClient.signOut()`

## Курсы валют

- **Источник:** OpenExchangeRates API (бесплатный тариф)
- **Расписание:** ежедневный cron в 6:00 UTC (`convex/crons.ts`)
- **Хранение:** таблица `rates` с индексом `by_date` (одна запись в день)
- **Фронтенд:** `useCurrenciesStore` получает последние курсы, запускает обновление если пусто
- **Fallback:** Если в базе нет курсов (новый деплоймент), курсы должны быть загружены через ежедневный cron или вручную через `npx convex run`
- **Использование:** `getAmountInRate()` конвертирует суммы в основную валюту для статистики

## Управление модальными окнами

Модальные окна используют локальные `ref` и паттерн `v-if` + `@close` emit. Видимость меню - модульный `ref` в `useMenuData.ts` (общий для всех вызовов). Остальные модалки (например, сортировка кошельков) используют локальные `ref` компонента. Глобального реестра модалок нет - они размонтируются автоматически при размонтировании родительских компонентов (например, при выходе хранилища сбрасываются в `null` → условие layout становится false → модалки исчезают).

## Персистентность сайдбара

Состояние показа/скрытия сайдбара на десктопе хранится в cookie (`finapp.isShowSidebar`), обеспечивая сохранение состояния между загрузками страниц.

## Bottom Sheet

Мобильная форма транзакции использует кастомный компонент `BottomSheet` с `useBottomSheetDrag` - ручная реализация drag-поддержки touch и mouse. Функции: настраиваемый порог начала закрытия, определение направления, затухание оверлея, учёт скролла (игнорирует drag при прокрутке контента), исключение sort-хэндлов.

## Логирование

`createLogger(prefix)` (`app/utils/logger.ts`) - логирование только для разработки. В продакшене работает только `.error()`. Все операции хранилищ и авторизации инструментированы с префиксами (`[wallets]`, `[trns]`, `[auth/middleware]` и т.д.). Проверяйте консоль браузера при `pnpm dev`.

## PWA

- Стратегия: `generateSW` из `@vite-pwa/nuxt` (без кастомного service worker)
- Прекеширование: все ассеты сборки кешируются автоматически
- `navigateFallback: '/'` отдаёт SPA-оболочку для всех навигаций (офлайн-поддержка)
- Runtime-кеширование: Google Fonts, иконки Iconify (CacheFirst)
- Start URL: `/dashboard`
- Манифест: `display: 'standalone'`

## Инструменты миграции

Бэкенд Convex включает внутренние мутации для миграции данных, вызываемые через `npx convex run`:

| Функция | Назначение |
|---------|-----------|
| `migrate:deleteUserData` | Удаление всех данных пользователя |
| `migrate:insertWallets` | Массовый импорт кошельков с ремаппингом ID |
| `migrate:insertCategories` | Массовый импорт категорий с ремаппингом parentId |
| `migrate:insertTrnsBatch` | Массовый импорт транзакций |
| `migrate:insertUserSettings` | Импорт настроек пользователя |

Скрипт миграции: `utils/migrate/firebase.js` - читает JSON-экспорт Firebase, трансформирует данные и вызывает мутации выше. Подробнее: [Миграция с Firebase](/ru/development/migration).
