Финапка прошла через два крупных изменения бэкенда. На этой странице они описаны как исторический контекст.

## Firebase -> Convex (v6 -> v7)

В Финапке v6 использовалась Firebase Realtime Database. Финапка v7 заменила её на Convex (backend-as-code, WebSocket-синхронизация).

Исходный код v6 хранится на ветке [`firebase`](https://github.com/ilkome/finapp/tree/firebase).

### Что изменилось

| Сущность | Firebase (v6) | Convex (v7) |
|----------|---------------|-------------|
| Счета | `accounts` со строковыми ключами | Таблица `wallets` с Convex ID |
| Категории | Строковые ID, `parentId` строкой | Convex ID, `parentId` как `Id<'categories'>` или `0` |
| Транзакции | Строковые ID, ссылки старыми ключами | Convex ID, все ссылки перемаплены |
| Настройки | `settings.lang` (`en`, `ru`) | `userSettings.locale` (`en`, `ru`) |
| Иконки | Формат `mdi mdi-xxx` | Формат `mdi:xxx` |

### Инструменты миграции (исторически - больше нет в репозитории)

Миграция была разовой операцией. Скрипты, которые её выполняли (`utils/migrate/firebase.js`, `convex/migrate.ts`), удалены из репозитория как неактуальные. Запускать их не нужно.

Скрипты читали JSON-экспорт из Firebase, трансформировали данные под схему Convex (ремаппинг ID, исправление формата иконок, нормализация настроек) и импортировали через внутренние мутации Convex батчами по 200.

## Convex -> Supabase+PowerSync (текущий бэкенд)

Финапка v7+ (Convex) была мигрирована на **Supabase Postgres + PowerSync** (offline-first синхронизация) с **Supabase Auth (email/password)**.

Это была разовая операция во время разработки, не повторяющийся пользовательский процесс:

1. Экспорт всех данных из деплоймента Convex командой `convex export --prod`.
2. Трансформация экспорта Python-скриптом, который сохранял значения `_id` из Convex как `text`-идентификаторы в Postgres (конвертация UUID не требовалась - id остались стабильными) и переопределял значения `userId` на соответствующие Supabase Auth uid.
3. Импорт через `psql` в экземпляр Supabase Postgres.

В схеме Postgres нет FK-ограничений, поэтому порядок импорта не имеет значения. RLS-политики и правила синхронизации PowerSync обеспечивают изоляцию данных по пользователям.
